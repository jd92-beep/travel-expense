# Trip Update AI Extraction — Deep Analysis & Improvement Plan

**Date:** 2026-06-11  
**Trigger:** User pasted an 8-day Jeju itinerary (tab-separated, with duration column, advice blocks) and the LLM extraction completely failed.

---

## Root Cause Analysis — Why Extraction Failed

### Failure Point 1: Local parser can't read tab-separated tables
**File:** `app-compact/src/lib/ai.ts:338-366` (`extractLocalDaySpots`)

The local parser has exactly two branches:
- **Pipe table** (L352): lines starting with `|` → split by `|`
- **Plain text** (L360): regex `^HH:MM name`

The user's format uses **tab characters** (`\t`) between columns:
```
06:30	抵達濟州機場	—
09:00	先食簡單早餐 / 咖啡	30–45分鐘
```
This matches NEITHER branch. The tab-separated rows are completely skipped, producing 0 spots from the local parser.

### Failure Point 2: Duration column not handled
Even if tabs were parsed, the third column contains durations (`30–45分鐘`, `約60–75分鐘車程`). The plain text regex (L360) captures everything after `HH:MM` as the spot name, so `先食簡單早餐 / 咖啡	30–45分鐘` would become a spot named `先食簡單早餐 / 咖啡	30–45分鐘` with the duration embedded.

### Failure Point 3: Advice blocks lost
**File:** `ai.ts:422` — `建議：` lines are filtered out:
```typescript
.filter((spot) => spot.name && !/建議[:：]/.test(spot.name))
```
The user's `建議：Day 1 唔好加正房瀑布...` is valuable context (day notes, skip conditions) but is discarded entirely.

### Failure Point 4: LLM timeout too aggressive for complex itineraries
**File:** `ai.ts:156-159`
```
TRIP_PRIMARY_TIMEOUT_MS = 8_000      // 8s per call
TRIP_FALLBACK_TIMEOUT_MS = 9_000     // 9s per call  
TRIP_FAST_LOCAL_DEADLINE_MS = 14_000 // 14s total
```
With a 28,000-char 8-day itinerary, a weak model needs 10-20s per call. Two stages × 8s = the model is killed mid-response. Since `hasFastLocalDraft` is false (local parser found 0 spots), the 14s deadline doesn't apply, but per-call timeouts still kill the LLM before it finishes.

### Failure Point 5: Two-stage LLM doubles latency
**File:** `ai.ts:928-946` — Stage 1 (organize) + Stage 2 (extract) = 2 API calls per model. For 6 models in the ladder, worst case = 12 API calls × 8-9s each = ~100s total. With the user's complex format, weak models may fail Stage 1, wasting the call.

### Failure Point 6: Organized itinerary truncation
**File:** `ai.ts:369` — `stringifyOrganizedItinerary` caps at 5000 chars. An 8-day itinerary with 10 spots each easily exceeds this, losing Day 5-8.

### Failure Point 7: No structured output enforcement
**File:** `workers/credential-broker/src/index.js:838-865` (Kimi), `867-886` (Mimo)
Only Google has `responseMimeType: 'application/json'`. Kimi/Mimo rely on system message `"Return strict JSON only"` which weak models often ignore, returning markdown-wrapped or partially-structured text.

## Improvement Plan

### Phase 1: Harden Local Parser (Highest Impact — works without any LLM)

#### Task 1.1: Add tab-separated table parsing
**File:** `app-compact/src/lib/ai.ts:338-366` (`extractLocalDaySpots`)

**Change:** Add a third branch for tab-separated rows, before the plain text branch:
```
if line contains \t and matches HH:MM pattern in first tab-field:
  split by \t
  time = first field (HH:MM)
  name = second field (place/activity)
  duration/notes = third field (optional)
```

**Specific logic:**
```typescript
// After pipe table branch (L358), before plain text branch (L360):
const tabCells = line.split('\t').map(c => c.trim()).filter(Boolean);
if (tabCells.length >= 2) {
  const timeMatch = tabCells[0].match(/^([01]?\d|2[0-3]):([0-5]\d)$/);
  if (timeMatch) {
    const time = `${timeMatch[1].padStart(2, '0')}:${timeMatch[2]}`;
    const name = tabCells[1];
    const duration = tabCells[2] || '';
    add(localSpotFromParts(time, name, rawLine, duration));
    continue;
  }
}
```

**Also update `localSpotFromParts`** (L323-336) to accept duration and store it in `timeEnd`:
- Parse `30–45分鐘` → calculate `timeEnd = time + 30min` (use the lower bound)
- Parse `約60–75分鐘車程` → extract numbers, calculate `timeEnd`
- Store duration text in `note` field if `timeEnd` calculation fails

**Verification:** Unit test with the user's exact Day 1 block → expect 10 spots with correct times and names.

#### Task 1.2: Extract duration as timeEnd
**File:** `app-compact/src/lib/ai.ts:323-336` (`localSpotFromParts`)

**Change:** Add `timeEnd` calculation from duration text:
```typescript
function parseDurationMinutes(text: string): number | null {
  const match = text.match(/(\d+)\s*(?:[–-]\s*\d+\s*)?分鐘/);
  return match ? Number(match[1]) : null;
}

function addMinutesToTime(time: string, minutes: number): string {
  const [h, m] = time.split(':').map(Number);
  const total = h * 60 + m + minutes;
  return `${String(Math.floor(total / 60) % 24).padStart(2, '0')}:${String(total % 60).padStart(2, '0')}`;
}
```

**Verification:** `09:00` + `30–45分鐘` → `timeEnd: "09:30"`. `11:15` + `60–75分鐘` → `timeEnd: "12:15"`.

#### Task 1.3: Capture advice blocks as day notes

**Change:** Instead of filtering `建議` lines, extract them as day-level notes:

**Verification:** Day 1 advice `唔好加正房瀑布。紅眼機到埗，容易攰...` appears in `day.highlight`.

#### Task 1.4: Improve spot name cleaning

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Tab parsing breaks existing pipe table format | Low | High | Separate branch, existing tests cover pipe tables |
| Increased timeouts cause slow UI feedback | Medium | Medium | Keep fast local draft as immediate feedback |
| Merge logic creates duplicate spots | Medium | Low | Dedup by `time + name` key |
| Single-stage extraction returns lower quality | Low | Medium | Keep two-stage as fallback for non-Google models |
| timeEnd calculation wrong for edge cases | Medium | Low | Use lower bound of duration range, show as estimate |
