# 2 Bug Fixes: Dropdown Position + Delete Account

## Bug 1: Trip Dropdown Opens on the Right

**Root Cause**: `Shell.tsx:63` — TripDropdown default align is `left-0`. In the dashboard header (line 419-431), the TripDropdown sits NEXT TO the trip name in a `flex items-center gap-2` row. The dropdown's `left-0` aligns with the LEFT edge of the chevron button, which is already to the RIGHT of the trip name. So the dropdown appears shifted right.

**Fix**: `Shell.tsx:425` — Pass `align="right"` to the TripDropdown so the dropdown's RIGHT edge aligns with the chevron's right edge, appearing directly below it.

**File**: `app-compact/src/components/Shell.tsx:425`
**Change**: Add `align="right"` prop to the TripDropdown on the dashboard header.

## Bug 2: Delete Account Has No Response

**Root Cause**: `Settings.tsx:955` — `supabaseAuth.deleteUserAccount()` calls `supabase.rpc('delete_own_user_account')`. If the RPC fails (e.g., function doesn't exist, network error, RLS issue), the error is caught at line 965-967 and `setStatus(...)` is called. But the modal stays open and the status message might not be visible (it's outside the modal). The user sees "no response".

**Likely cause**: The `delete_own_user_account` RPC might not be deployed to the Supabase project, or it requires specific permissions. The error is caught but not prominently displayed.

**Fix**:
1. Add visible error feedback inside the modal (not just the status bar)
2. Add a loading indicator while the deletion is in progress
3. After successful deletion, explicitly navigate to login (don't rely solely on auth state change)
4. Add a try-catch around `signOut()` separately so even if the RPC succeeds but signOut fails, the user gets feedback

**File**: `app-compact/src/tabs/Settings.tsx:928-967`
**Changes**:
- Show error message inside the modal dialog
- Add explicit `window.location.reload()` or state reset after successful deletion to force redirect to login
- Ensure `setBusy(null)` is called in the finally block
